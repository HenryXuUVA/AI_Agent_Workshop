## Questions and Answers

### What features of Python does `ToolNode` use to dispatch tools in parallel? What kinds of tools benefit most from parallel dispatch?

`ToolNode` uses Python concurrency features to run multiple tool calls at the same time, including an executor for synchronous work and `asyncio.gather()` for asynchronous work. This is most useful for independent, I/O-bound tools like API calls, database queries, search, and retrieval tasks.

### How do the two programs handle special inputs such as `verbose` and `exit`?

`toolnode_example.py` does not handle `verbose` or `exit` specially because it only runs preset test queries and is not interactive. `react_agent_example.py` explicitly checks for `exit`, `verbose`, and `quiet`, using `exit` to end the graph and the others to toggle debug output without sending them to the agent as normal messages.

### Compare the graph diagrams of the two programs. How do they differ?

The diagrams in Topic 4 are both generated by `react_agent_example.py`, and they show two different layers of that one program. One graph is the small internal ReAct loop (`agent <-> tools`), while the other is a larger wrapper graph that adds input handling, output, history trimming, and command routing.

### When is the LangChain ReAct agent structure too restrictive, making `ToolNode` a better approach?

A `ToolNode` approach is better when you need custom orchestration instead of the fixed ReAct pattern of model, tools, then model again. For example, if you want to fan out several tool calls in parallel, retry only failures, and merge results through custom routing logic, the ReAct agent structure becomes too restrictive.
